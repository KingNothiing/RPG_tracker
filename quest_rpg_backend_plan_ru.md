# План Backend-проекта Quest RPG

## Рабочее название

- QuestRPG API
- Habit Kingdom API
- Discipline Engine

Сначала используй временное название. Переименовать можно позже, когда концепция станет точнее.

## Идея продукта

Это не todo-приложение с очками.

Это backend для персональной RPG-системы, в которой пользователь:

- создаёт привычки и квесты;
- выполняет ежедневные и одноразовые действия;
- получает опыт и повышает уровень;
- накапливает серии выполнения;
- открывает достижения;
- получает награды или штрафы;
- развивает персонажа через систему правил.

Техническая цель проекта: показать настоящее backend-мышление:

- моделирование предметной области;
- бизнес-правила;
- безопасность транзакций;
- проектирование API;
- фоновые задачи;
- тестирование критической логики;
- понятную эволюцию проекта от MVP к более production-like состоянию.

## Почему этот проект сильный для портфолио

- Он запоминается лучше, чем обычный трекер, блог или магазин.
- В нём достаточно бизнес-логики, чтобы обсуждать проект на собеседованиях.
- Его можно начать с малого и развивать месяцами.
- У него есть реалистичный путь к монетизации.
- Он позволяет объяснить, зачем была добавлена каждая технология.

## Чем он отличается от базового habit tracker

Базовый habit tracker:

- создать привычку;
- отметить выполнение;
- посмотреть серию.

Этот проект:

- использует игровые механики как модель предметной области;
- хранит события и историю прогрессии;
- вычисляет награды и штрафы через правила;
- открывает достижения по условиям;
- поддерживает запланированные сбросы и отложенные действия;
- позже может поддерживать сезоны, лиги, инвентарь и premium-механику.

## Главная продуктовая фантазия

Пользователь не просто "отслеживает задачи". Пользователь развивает персонажа.

Примеры:

- "Прочитать 20 страниц" даёт XP интеллекта.
- "Тренировка" даёт XP силы.
- Пропуск критического квеста накладывает штраф.
- Серия в 7 дней открывает значок.
- Достижение 5 уровня открывает новый узел навыков.

Такая рамка делает проект эмоционально интересным и технически насыщенным.

## Целевая аудитория

Основная:

- студенты;
- люди, занимающиеся саморазвитием;
- пользователи, которым нравится продуктивность с игровыми механиками.

Вторичная:

- создатели, которым позже могут понадобиться геймифицированные сообщества;
- Telegram- или mobile-пользователи, если ты расширишь проект.

## Ключевые принципы

- Backend first.
- Frontend на старте необязателен.
- Каждая добавленная технология должна решать реальную проблему.
- MVP должен быть достаточно маленьким, чтобы его можно было закончить.
- Этапы роста должны добавлять по одному обучающему слою за раз.

## Объём MVP

Первая версия должна доказать основной цикл:

1. Пользователь регистрируется и входит в систему.
2. Пользователь создаёт привычки и квесты.
3. Пользователь отмечает выполнение.
4. Система выдаёт XP и обновляет серии.
5. Пользователь видит прогресс персонажа и историю событий.

Если этот цикл работает хорошо, проект уже становится достойным портфолио.

## Функции MVP

- регистрация пользователя и аутентификация;
- профиль персонажа;
- создание привычек;
- создание ежедневных и одноразовых квестов;
- endpoint завершения квеста;
- расчёт XP и уровня;
- отслеживание серий;
- журнал событий;
- простая страница статистики через API;
- админка для проверки данных.

## Что явно исключено из MVP

Не нужно делать это в первую очередь:

- инвентарь;
- дерево навыков;
- гильдии;
- лидерборды;
- социальные функции;
- платежи;
- Telegram-бот;
- WebSocket;
- AI-функции.

Это этапы расширения, а не стартовые требования.

## Почему такой порядок стека правильный

- Django и DRF учат структуре и базовым принципам API.
- PostgreSQL учит работать с реальной базой данных.
- pytest учит уверенности в коде и защите от регрессий.
- Docker делает проект профессиональнее, не меняя бизнес-логику.
- Celery и Redis действительно становятся нужны только тогда, когда появляются запланированные задачи и фоновая обработка.

Такой порядок защищает от "stack cosplay", когда инструменты добавляют просто ради солидного вида.

## Связи

- User -> один CharacterProfile
- User -> много Habits
- User -> много Quests
- Quest -> опционально связан с Habit
- Quest -> много записей Completion только если модель допускает повторные попытки, иначе одна запись completion
- User -> много записей EventLog
- User -> много достижений через UserAchievement

## Бизнес-правила MVP

Это сердце backend-части.

### Правило XP

- Каждый квест даёт фиксированную награду XP.
- Сложность может модифицировать XP.
- XP выдаётся только один раз за допустимое завершение.

### Правило уровня

Сначала держим просто:

- уровень 1 начинается с 0 XP;
- каждые 100 XP дают новый уровень.

Позже можно перейти к нелинейной формуле.

### Правило дневной серии

- Если пользователь завершил хотя бы один daily-квест за день, серия сохраняется.
- Пропуск дня сбрасывает серию.
- Рекордная серия сохраняется отдельно.

### Правило достижений

Начни с 3-5 простых достижений:

- первый завершённый квест;
- серия в 3 дня;
- достижение 3 уровня;
- завершение 10 квестов;
- создание 5 привычек.

### Правило журнала событий

Каждое важное изменение состояния должно создавать событие.

Это даёт:

- более простую отладку;
- объяснимую прогрессию;
- источник для будущей аналитики;
- хорошие темы для обсуждения на собеседовании.

## Важные инженерные ограничения

### Idempotency

Endpoint завершения квеста не должен случайно выдавать XP дважды.

Нужно явно защититься от:

- повторных клиентских запросов;
- повторной отправки формы после обновления страницы;
- двойных кликов;
- будущих race conditions.

### Transactions

Когда квест завершается, эти обновления должны рассматриваться как единое целое:

- отметить квест как завершённый;
- создать запись completion;
- выдать XP;
- обновить уровень;
- обновить серию при необходимости;
- создать события;
- открыть достижения, если условия выполнены.

Используй транзакции базы данных, чтобы частичные обновления не оставляли систему в сломанном состоянии.

### Permissions

Пользователь должен иметь доступ только к своим привычкам, квестам, профилю и событиям.

Это базовая, но важная backend-гигиена.

## Разделение ответственности между приложениями

### accounts

- аутентификация;
- создание пользователей;
- сериализаторы, связанные с пользователем.

### characters

- модель профиля;
- read-models для уровня и XP;
- endpoints профиля.

### habits

- CRUD привычек.

### quests

- CRUD квестов;
- входная точка endpoint-а завершения.

### achievements

- определения достижений;
- записи об открытии достижений.

### events

- хранение журнала событий;
- endpoints истории в стиле audit log.

### progression

- чистые бизнес-сервисы:
  - grant_xp;
  - update_level;
  - update_streak;
  - unlock_achievements.

Такое разделение хорошо смотрится на собеседованиях, потому что ты можешь чётко объяснить, где живёт бизнес-логика.

## Идея service layer

Не прячь всю логику в serializers или views.

Используй application services, например:

- `complete_quest(user, quest_id)`
- `grant_xp(profile, amount)`
- `check_achievements(user)`

Преимущества:

- лучшее тестирование;
- более тонкий API-слой;
- более простой дальнейший рост;
- более чистое объяснение на собеседовании.

## Стратегия тестирования

Для junior-портфолио тесты дают большое преимущество.

Минимальные группы тестов:

- тесты аутентификации;
- тесты permissions;
- тесты завершения квестов;
- тесты XP и уровней;
- тесты серий;
- тесты открытия достижений;
- тесты защиты от повторного completion.

Ключевая мысль:

Тебе не нужно гигантское покрытие. Тебе нужно хорошее покрытие рискованной логики.

## Лучшее первое расширение после MVP

Выбери ровно один вариант:

1. Движок достижений
2. Инвентарь и награды
3. Ежедневный планировщик на Celery
4. Интерфейс через Telegram-бота

Рекомендуемое первое расширение:

- движок достижений

Причина:

- очень заметно для пользователя;
- много backend-логики;
- низкая инфраструктурная стоимость.

## Пути монетизации

- premium-темы или продвинутые системы прогрессии;
- платные community-челленджи;
- платная аналитика;
- premium-функции Telegram-бота;
- white-label движок геймификации для создателей;
- командный режим для небольших групп.

Не нужно строить монетизацию в первую очередь. Просто держи домен открытым для неё.

## Что говорить на собеседованиях

Хорошая формулировка:

- "Я специально выбрал backend-heavy проект с нетривиальной моделью предметной области."
- "Я разделил API-слой и логику прогрессии."
- "Я использовал транзакции, чтобы completion, XP, series и achievements обновлялись согласованно."
- "Я добавил event logging, чтобы изменения состояния были объяснимыми и тестируемыми."
- "Я ввёл Celery только тогда, когда действительно появились задачи по расписанию."

Это звучит намного сильнее, чем "Я сделал CRUD-приложение".

## Частые ошибки, которых стоит избегать

- Делать MVP слишком большим.
- Слишком рано добавлять frontend.
- Подключать Celery до появления реальной необходимости в задачах по расписанию.
- Складывать всю логику во views или serializers.
- Строить social-функции до того, как станет надёжным single-user цикл.
- Добавлять слишком много игровых механик до того, как базовая прогрессия станет приятной.

## Практический следующий шаг

Начни только с Milestone 0 и Milestone 1.

Не проектируй всю вселенную в коде в первый же день.

Построй:

- структуру репозитория;
- Django-проект;
- auth;
- профиль персонажа.

Затем остановись и проверь базу, прежде чем переходить к привычкам и квестам.

## Если ты строишь это с Codex

Используй Codex как:

- помощника по scaffolding;
- ревьюера архитектурных решений;
- объясняющего ассистента по внутренностям Django;
- помощника в написании тестов;
- ассистента по рефакторингу.

Не используй его только как кнопку "сделай всё за меня".

Для каждого milestone задавай вопросы:

- какую проблему решает этот код;
- почему эта модель устроена именно так;
- почему этот endpoint находится здесь;
- что может сломаться;
- как это нужно тестировать.

Так проект становится одновременно и портфолио, и обучением.
